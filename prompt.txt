**Create a high-performance native C++ IRC chat rendering control for WPF using Direct2D/DirectWrite interop via D3DImage.**

**Goal:** Build a console-style chat control for a WPF application that can sustain **1,000–5,000+ lines per second** with a **50,000-line scrollback buffer** while maintaining sub-16ms frame times and a minimal memory footprint (~30 MB for the full buffer).

**Architecture Overview:**
- **WPF Host (C#):** A custom `D3DImage`-derived class that hosts a shared DirectX surface. It should expose a simple API (`AddLine(string text)`, `ScrollToEnd()`, `Clear()`) and use a `DispatcherTimer` (~60 FPS) to invalidate dirty regions.
- **Native Renderer (C++ DLL):** A C++ DLL compiled with `/clr` or pure native with C exports. It owns a Direct2D render target and DirectWrite text renderer. It must expose C-style exports: `CreateRenderer`, `DestroyRenderer`, `AddLine`, `RenderFrame`, `GetSharedHandle`, `SetSize`, `ScrollBy`, `ScrollToLine`.

**Core Data Structure (Zero-Allocation Hot Path):**
- Implement a **fixed-size ring buffer** (capacity: 50,000 lines) with pre-allocated `LineSlot` structs. No `std::string`, no heap allocation, no exceptions on the hot path.
- Each `LineSlot` should contain:
  - A fixed `char` text buffer (e.g., 512 bytes) or a pooled buffer for longer lines.
  - A segment array (`uint16_t` pairs or packed structs) representing IRC color/format runs.
  - Metadata: timestamp, line length, segment count.
- When the buffer is full, overwrite the oldest slot. Maintain head/tail indices.

**IRC Parsing (mIRC Formatting):**
- Parse control codes inline: `\x02` (bold), `\x03` (color: `fg[,bg]`), `\x0F` (reset), `\x1F` (underline), `\x16` (italic).
- The parser must run in the native DLL, fed by a **lock-free MPSC queue** (e.g., moodycamel::ConcurrentQueue or a custom circular buffer with atomics) so the network thread can push lines without blocking the render thread.
- Parse results should be written directly into the next available `LineSlot` in the ring buffer. No temporary string objects.

**Rendering Strategy:**
- Use **Direct2D** for background fills/rects and **DirectWrite** for glyph rendering.
- **Do NOT create an `IDWriteTextLayout` per line.** Instead, use `IDWriteTextFormat` + `DrawText` (or `DrawGlyphRun`) per colored segment. Cache the `IDWriteTextFormat` and `IDWriteFontFace`.
- Render only the **visible viewport**. Calculate `startLine` from scroll offset and `lineHeight`. Stop rendering when `y > clipHeight`.
- Implement **batch invalidation**: collect incoming lines for 1–2ms before triggering a render pass to avoid per-line redraw storms.
- Use **dirty rect tracking**: if only new lines arrive, only the bottom region needs redraw; if scrolling, use scroll-bit blit where possible.
- Support a configurable color palette (at least 16 mIRC colors) mapped to `D2D1::ColorF`.

**WPF Interop (D3DImage):**
- The C++ DLL should create a D3D11 device with a BGRA texture (or D3D9Ex surface for `D3DImage` compatibility), render into it, and expose the shared texture handle.
- The C# `D3DImage` wrapper should:
  - Call `SetBackBuffer(D3DResourceType.IDirect3DSurface9, sharedHandle)` or equivalent.
  - Lock the image, call `AddDirtyRect()` covering the updated region, and unlock.
  - Forward `MouseWheel`, `SizeChanged`, and `KeyDown` events to the native DLL via P/Invoke to handle scrolling and selection.
- The native surface should blend cleanly with WPF UI (e.g., appear inside a `Border` with WPF chrome around it).

**Memory & Performance Constraints:**
- **Hot path (line arrival):** Zero heap allocation. Only atomic queue pushes and ring buffer writes.
- **Hot path (render):** No `std::vector` resizes, no string copies. Use pre-allocated glyph metrics caches if needed.
- Target memory: ~30 MB for the 50,000-line buffer. The GPU texture is a fixed cost based on control size.
- Avoid WPF `FlowDocument`, `Run`, `Span`, or any managed text objects for chat content entirely.

**Deliverables:**
1. A **Visual Studio solution** with two projects:
   - `IrcRendererNative`: A C++ DLL project with Direct2D/DirectWrite/D3D11 setup, ring buffer, IRC parser, and C exports.
   - `IrcChatWpf`: A C# WPF application with the `D3DImage` host control and a test harness that simulates high-throughput IRC spam (inject 1,000+ lines/sec with random color codes).
2. A `README` explaining the build dependencies (Windows SDK, VC++ redist), the P/Invoke contract, and how to tune the ring buffer size and batch timing.
3. Basic features: vertical scrolling, auto-scroll to bottom, manual scrollback, and a visual scrollbar that maps to the native ring buffer's logical range.

**Explicitly Do NOT:**
- Use C++/CLI `UserControl` with WPF layout for the text content.
- Use `std::string`, `std::stringstream`, or `std::vector` push_backs on the network/render hot path.
- Render off-screen lines or create per-line DirectWrite layouts.

**Context:** This replaces a WPF `FlowDocument`-based IRC control that collapses under high message volume due to managed object allocation, GC pressure, and retained-mode visual tree overhead. The native control must bypass all WPF text rendering to achieve the throughput and memory targets.
